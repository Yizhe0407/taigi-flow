# ADR-002: ASR 採用雙 Backend 抽象介面（Qwen3-ASR + Breeze-ASR-26）

- **狀態**：Accepted
- **日期**：2026-04-18
- **決策者**：[專題團隊]

## 背景

台語 ASR 模型選擇不多，目前確定原生支援台語的兩個選項各有明顯優劣：

- **Qwen3-ASR**：多語言 ASR，Qwen3-Omni 基座，支援 22 種中文方言含閩南語。**原生串流推論**（vLLM backend，TTFT 約 92ms）。有 0.6B / 1.7B 兩種規格。
- **Breeze-ASR-26 (BreezyASR-Taigi)**：Whisper-large-v2 微調，約 10,000 小時台語合成語料專訓。**無原生串流**，需等整段語音再辨識。

在選型時面臨「串流能力 vs 台語專精度」的明確取捨，且兩者都有值得嘗試的價值。

## 決策

**不鎖定單一模型**，而是：

1. 設計 `BaseASR` 抽象介面
2. 實作兩個具體類別：`Qwen3ASR`、`BreezeASR26`
3. 透過環境變數或設定切換
4. Phase 2 結束時以實測數據決定**預設 backend**
5. 保留兩個實作在程式碼中，供未來切換

## 理由

1. **介面抽象本來就該做**：即使只用一個模型，也應該用介面隔離。雙實作讓抽象有「被驗證」的價值（單一實作的介面常常意外耦合某個模型的特性）。

2. **紙上選型不可靠**：模型卡片宣稱的準確率 vs 實際場景表現常有差距。必須以真實目標場景（公車路線詢問、使用者口音）跑過才能決定。

3. **對書面報告加分**：多一個「ASR 評估實驗」章節，顯著提升方法論完整度。「我們比較了 X 與 Y，在 ___ 指標上選擇 ___」遠比「我們用了 X」有說服力。

4. **風險對沖**：若預設 backend 在 demo 當天出問題（網路、模型當機），可切換備援。

5. **兩者皆為開源**：都用 Apache 2.0 授權，無商業風險。

## 替代方案

### 只選 Qwen3-ASR
**優點**：延遲低（TTFT 92ms）、官方支援完整、開發單一流程
**缺點**：台語準確度未必最優、無備援

### 只選 Breeze-ASR-26
**優點**：台語訓練資料最多、辨識率預期最高
**缺點**：無串流，使用者感知延遲比 Qwen3 多 300–800ms；對即時對話場景是致命缺陷

### 使用 Whisper (台語微調社群版本)
**優點**：生態成熟
**缺點**：品質參差、無官方支援、對中英台混用情境弱

## 評估計畫

於 Phase 2 結束時執行，結果寫入 `docs/asr_evaluation.md`：

| 指標 | 量測方式 | 通過門檻 |
|------|---------|---------|
| CER（字元錯誤率） | 錄製 30 段測試語音，人工轉譯比對 | 兩者差距 < 10% 即可並列候選 |
| 平均 TTFT | 最後音訊 chunk 送出到首個辨識結果 | < 500ms 為佳 |
| 端到端延遲 | 使用者說完到 ASR final 產出 | < 1000ms 為佳 |
| GPU 記憶體峰值 | nvidia-smi 監測 | 單卡 < 12GB |

**決策準則**：
- CER 差距 < 5% → 選延遲較低者（預期為 Qwen3-ASR 0.6B）
- CER 差距 > 10% → 選 CER 較低者，接受延遲代價
- 5–10% 之間 → 綜合考量，以延遲為主

## 後果

✅ **正面**：
- 實測驅動的選型，決策可信
- 書面報告有實驗章節
- 風險對沖

⚠️ **負面**：
- Phase 2 工作量增加（要包裝兩個模型）
- GPU 資源需求增加（若並行載入，雖然實際運行只用一個）
- 代碼維護成本略高
