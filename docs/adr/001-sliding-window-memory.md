# ADR-001: 採用滑動視窗記憶，不使用摘要壓縮

- **狀態**：Accepted
- **日期**：2026-04-18
- **決策者**：[專題團隊]

## 背景

Agent 需要在對話中保持上下文記憶，讓 LLM 能回應「你剛才問的 307 公車」這類指涉。有三種常見方案：

1. **滑動視窗 (Sliding Window)**：只保留最近 N 輪對話，超過丟棄
2. **摘要壓縮 (Summary Compression)**：每 N 輪後讓 LLM 把舊對話摘要成一段文字
3. **混合式 (Hybrid)**：近期原文 + 遠期摘要

需要在「記憶完整度」「延遲」「實作複雜度」「記憶體安全」之間取捨。

## 決策

採用方案 1：**滑動視窗**，`max_turns = 12`（可調）。

## 理由

1. **記憶體有上界且可證明**：視窗大小固定，空間複雜度 O(1)。12 輪 × 2 訊息 × 150 tokens × 4 bytes ≈ 14 KB。**無論對話多長都不會爆**，這是數學上可保證的結論，書面報告可直接引用。

2. **無額外延遲**：摘要方案每 N 輪要多呼叫一次 LLM 做摘要，對即時語音對話來說是不可接受的延遲插入。

3. **使用情境匹配**：本系統是單 session、5–15 分鐘的短對話、每輪 50–150 tokens。12 輪視窗已遠超典型對話長度，實際極少會觸及視窗邊界。

4. **實作簡單**：10 行程式碼，可完整單元測試。摘要方案涉及提示工程、摘要品質衰減、成本管理。

## 替代方案

### 摘要壓縮
**優點**：保留長期記憶、上下文「感覺」更連貫
**缺點**：
- 每次摘要多一次 LLM 呼叫，增加延遲與 API 成本
- 摘要品質隨次數衰減（摘要的摘要）
- 對專題短對話場景是過度工程

### 混合式
**優點**：最接近人類記憶模式
**缺點**：兼具摘要方案的所有缺點，且實作最複雜

### 不做任何限制（全保留）
**優點**：零實作成本
**缺點**：長對話會觸及 LLM context window 上限，最終仍會爆

## 後果

✅ **正面**：
- 記憶體絕對安全（可數學證明）
- 實作與測試簡單
- 延遲最低
- 書面報告可明確交代空間複雜度

⚠️ **負面**：
- 超過 12 輪後早期對話會「遺忘」（對本場景幾乎不觸發）
- 若未來真的需要長期記憶（例如記住使用者姓名），需要另做機制
