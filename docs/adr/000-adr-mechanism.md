# ADR-000: 採用 Architecture Decision Records (ADR) 機制

- **狀態**：Accepted
- **日期**：2026-04-18
- **決策者**：[專題團隊]

## 背景

專案進行中會有大量技術選型與架構決策。這些決策在當下有明確理由，但半年後 review 時常常忘記「為何不選 X」，導致：
1. 後人（或未來的自己）重新討論同樣問題
2. 實作時因不理解原決策而做出與設計衝突的改動
3. 書面報告難以交代決策歷程

## 決策

採用輕量 ADR 格式記錄重要決策。每份 ADR 是 `docs/adr/` 下的一個 markdown 檔。

### 什麼該寫成 ADR

- 技術選型（選用 X 而不用 Y）
- 架構權衡（為何拆/合這個模組）
- 範圍決策（為何不做某功能）
- 跨模組的約定（命名、格式、流程）

### 什麼不用寫 ADR

- 單純的實作細節（變數名、私有函式結構）
- 已被上一份 ADR 涵蓋的衍生決策
- 明顯的行業最佳實踐（用 HTTPS、不存密碼明文等）

## 格式

每份 ADR 檔名格式：`NNN-短標題-用-kebab-case.md`，例如 `001-sliding-window-memory.md`。

內容必須包含以下章節：

```markdown
# ADR-NNN: 標題

- **狀態**：Proposed | Accepted | Deprecated | Superseded by ADR-XXX
- **日期**：YYYY-MM-DD
- **決策者**：[誰]

## 背景
描述問題情境、為何需要做這個決策。

## 決策
具體選擇了什麼。

## 理由
為何這樣選。至少列出 2-3 個理由。

## 替代方案
討論過哪些其他選項，為何沒選。

## 後果
這個決策帶來的正面與負面影響。

## 相關文件
- 連到 plan.md 的相關章節
- 連到相關 task
- 連到前置或後續 ADR
```

## 狀態定義

- **Proposed**：提案中，尚未採納
- **Accepted**：已採納並執行中
- **Deprecated**：已不再適用，但暫未被新決策取代
- **Superseded by ADR-XXX**：已被新決策取代

**一旦 ADR 進入 Accepted**，不再修改內容（除了修正 typo）。若決策改變，開新 ADR 並將舊的標為 Superseded。

## 理由

- **輕量**：每份 ADR 幾十到幾百行，寫作負擔低
- **去中心化**：每個決策是獨立檔案，避免單一大文件爆炸
- **時間順序清楚**：透過編號自然呈現決策演進
- **Git 友善**：純文字檔，可以 diff、可以在 PR 中 review

## 替代方案

- **寫進 plan.md**：缺點是 plan 會越來越長、歷史決策與現行設計混雜
- **Confluence / Notion**：增加工具負擔，與程式碼不同步
- **完全不記錄**：最差，半年後所有人都忘了為什麼

## 後果

✅ **正面**：決策可追溯、新人 onboarding 容易、書面報告有現成素材
⚠️ **負面**：需要紀律維護，懶得寫就失去價值
